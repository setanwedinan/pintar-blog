   - Returns ISO 8601 format: `YYYY-MM-DDTHH:MM:SSZ`

2. **Added post-processing validation:**
   - Before saving to file, scripts now check pubDate format
   - If pubDate lacks time (doesn't contain 'T'):
     - Parse the date
     - Add proper time (12:00 UTC)
     - Replace in content
   - If pubDate is missing entirely:
     - Add default pubDate with proper time

**Validation Logic:**
```javascript
// Check if pubDate exists and has time
const pubDateMatch = markdownContent.match(/^pubDate:\s*([^\n\r]+)/m);

if (pubDateMatch) {
  const currentPubDate = pubDateMatch[1].trim();

  // Check if pubDate has time (contains 'T')
  if (!currentPubDate.includes('T')) {
    // Add proper time
    const date = new Date(currentPubDate);
    date.setUTCHours(12, 0, 0, 0);
    const properPubDate = date.toISOString();
    markdownContent = markdownContent.replace(pubDateRegex, `pubDate: ${properPubDate}`);
  }
}
```

## USDIDR Rate Stale Issue - Multiple Sources Solution

**Problem:** Faizal noticed USD/IDR rate stayed at 16,812 for 4 consecutive days (14, 16, 17, 18 Feb 2026). Rate seemed stuck.

**Root Cause:** BCA e-Rate doesn't update at fixed time (08:00 WIB). Cron runs at 08:00, but BCA sometimes updates later (09:00, 10:00, etc.). This caused stale data in reports.

**Solution Implemented:**

1. **Multiple Data Sources with Fallback Logic:**
   - Primary: BCA e-Rate (local Indonesian bank)
   - Fallback 1: Frankfurter API (ECB data)
   - Fallback 2: ExchangeRate-API

2. **Stale Detection Logic:**
   - Checks if BCA rate is same for 2+ consecutive days
   - If stale (≥2 days), automatically falls back to Frankfurter API
   - If Frankfurter fails, tries ExchangeRate-API
   - If all fail, uses last known rate with warning

3. **Buy/Sell Spread Estimation:**
   - API sources only provide mid-rate
   - Estimated buy/sell: mid-rate ±0.25% (typical bank spread)

4. **History Update Fix:**
   - Script now updates existing entries instead of only appending
   - Prevents stale data persisting in history.json

5. **Git Error Handling:**
   - Added try-catch for git commit/push
   - Doesn't fail whole script if git fails (e.g., no changes detected)

**Test Results:**
- Manual test showed BCA now updating: 16,837 (buy: 16,792 / sell: 16,882)
- Fallback logic working: Frankfurter API returns 16,829 when BCA stale
- History update working: Correctly updates existing date entries

## Blog Post Ordering Issue - pubDate Format Fix

**Problem:** Faizal noticed blog posts were not ordered correctly by posting time. Posts created on same day appeared in wrong order.

**Root Cause:** In Astro, blog posts are sorted by `pubDate`. The format was inconsistent:
- Some had full ISO 8601 with time: `2026-02-16T01:00:00Z` ✓
- Some only had date: `2026-02-17` ✗

When `pubDate` lacks time, Astro defaults to 00:00 UTC (or another default), causing incorrect ordering for posts on the same day.

**Example Issues (17 February):**
- usdidr-report: `pubDate: 2026-02-17` (no time)
- aibubble-dashboard: `pubDate: 2026-02-17` (no time)

Both should have been created at 08:00 WIB (01:00 UTC), but without time, they defaulted incorrectly.

**Solution Implemented:**

1. **Added `getPubDate()` function to usdidr-report script:**

   - Sets time to 01:00 UTC (08:00 WIB for daily reports)
   - Returns ISO 8601 format: `YYYY-MM-DDTHH:MM:SSZ`

2. **Updated Blog Post Generation:**
   - Changed `pubDate: ${current.date}` to `pubDate: ${getPubDate(current.date)}`
   - Now generates proper format: `2026-02-18T01:00:00.000Z`

3. **Updated Cron Job Payloads:**
   
   **a) pintarblog-daily-auto (21:00 WIB):**

   **b) AI Bubble Daily Dashboard (08:00 WIB):**

   **c) daily-usd-idr-report (08:00 WIB):**
